# ADR-001: Aggregation source for re-launch and user-question counts

- Status: proposed
- Change: add-change-cost-telemetry
- Date: 2026-07-04

## Context

REQ-agents-001 fixes the archive report's observable Cost block (tokens/phase, re-launches,
user questions) but explicitly defers the internal aggregation *mechanism* to design. The
§9 harness metrics must become derivable per change without adding new hook scope or new
`state.yaml` schema.

## Decision

- **Tokens per phase**: group `phase-costs.jsonl` records by `phase`, sum `est_tokens`.
- **Re-launches per phase**: `count(records for phase) - 1` (floored at 0), derived purely
  from `phase-costs.jsonl` — one record equals one dispatch, so a repeat is a second row.
- **User questions**: sum of integer `phases.*.questions_asked` in `state.yaml` (missing → 0).

## Alternatives

- Dedicated re-launch counter in `state.yaml` — redundant; the per-dispatch JSONL already
  encodes it, and a separate counter can drift from the ground-truth rows.
- Emit synthetic "question" events into `phase-costs.jsonl` from the hook — rejected: the
  `SubagentStop` hook has no visibility into orchestrator-asked questions, so it would have
  to fabricate them and widen its scope.

## Consequences

Re-launch and token figures share one file and one aggregation pass (consistent, cheap).
Questions stay owned by the orchestrator via the existing `questions_asked` field, so the
hook stays minimal. Reversible: aggregation lives only in the `sdd-archive` skill; changing
the source later touches no persisted data model. If a phase's `questions_asked` is never
written, its questions read as 0 — an undercount tolerated because figures are labeled
estimated/best-effort.
