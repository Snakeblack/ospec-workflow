# VS Code Agent Plugin Installation

This repository is installed as a VS Code Agent Plugin. It is not a VS Code extension workflow.

Agent Plugins are preview functionality. Before installing, review the plugin contents because plugins can include hooks and MCP servers that execute code locally.

## What this plugin provides

The plugin manifest is `.plugin/plugin.json`. It declares the workflow bundle that VS Code loads when the plugin is enabled.

| Area | Source | What it provides |
| --- | --- | --- |
| Agents | `agents/` | SDD orchestration and phase agents. |
| Commands | `commands/` | User-facing command entrypoints such as `/sdd-new`, `/sdd-apply`, and `/sdd-verify`. |
| Skills | `skills/` | Reusable working rules for SDD phases, reviews, commits, docs, and related workflows. |
| Instructions | `rules/` | Plugin-bundled instruction files generated by the VS Code plugin creation flow. |
| MCP servers | `.mcp.json` | Context7 MCP server configuration using `CONTEXT7_API_KEY`. |
| Hooks | `hooks.json` and `scripts/hooks/` | Local PowerShell hook scripts for session persistence, tool-use validation, and OpenSpec artifact checks. |

The `.github/instructions/` directory is only a workspace mirror for instruction files. The plugin-bundled rules live in `rules/`.

## How to install from source

Use this path when you want VS Code to manage the plugin from a repository source.

1. Open a VS Code build that supports Agent Plugins preview.
2. Open the Agent Plugins install flow.
3. Choose the option to install from a Git repository URL.
4. Enter the repository URL for this project.
5. Review the plugin manifest, hooks, MCP server configuration, and scripts before accepting the install.
6. Enable the plugin when VS Code prompts you.
7. Reload VS Code if the plugin does not appear immediately.

Trust checklist before accepting:

- Confirm `.plugin/plugin.json` points only to expected plugin assets.
- Review `hooks.json` because it starts local PowerShell scripts.
- Review `scripts/hooks/` because those scripts run on hook events.
- Review `.mcp.json` because it can start local processes for MCP servers.
- Confirm you are comfortable providing `CONTEXT7_API_KEY` when MCP access is needed.

Plugin support may be disabled by organization policy. If the Agent Plugins UI or settings are missing, check your VS Code build and your organization-managed settings.

## How to enable locally with `chat.pluginLocations`

Use this path when you are developing or testing this repository directly from disk.

1. Clone the repository locally.
2. Open VS Code settings as JSON.
3. Add this repository root to `chat.pluginLocations`.
4. Reload VS Code.
5. Open the Agent Plugins view and confirm `ospec-workflow` is enabled.

Minimal `settings.json` example:

```json
{
  "chat.pluginLocations": [
    "C:\\dev\\Hiberus\\ospec-workflow"
  ]
}
```

If you keep the repository somewhere else, replace the path with your local checkout path.

## How to verify that agents and skills loaded

Start with the visible entrypoints, then inspect details only if something is missing.

| Check | Expected result |
| --- | --- |
| Agent Plugins view | `ospec-workflow` is listed and enabled. |
| Chat agent picker | `sdd-orchestrator` is available as the user-facing SDD agent. |
| Command entrypoints | `/sdd-new`, `/sdd-lite`, `/sdd-continue`, `/sdd-apply`, `/sdd-verify`, and `/sdd-archive` are available. |
| Plugin details | Agents, commands, skills, MCP servers, and hooks are shown from `.plugin/plugin.json`. |

Smoke test:

```text
Ask the SDD orchestrator: summarize the available SDD workflow commands without creating files.
```

Expected behavior:

- The orchestrator explains the SDD flow.
- It does not expose phase agents as normal user commands.
- It can reference skills and shared rules when needed.
- It does not create OpenSpec artifacts for a read-only summary request.

If skills do not appear to load, confirm the plugin is enabled and that the repository contains `skills/`. This repository currently has no `.atl/skill-registry.md`; that is expected for this task and should not block plugin loading.

## How to verify MCP server availability

The plugin declares MCP configuration in `.mcp.json`.

| MCP server | Purpose | Required input |
| --- | --- | --- |
| `io.github.upstash/context7` | Context7 library documentation lookup. | `CONTEXT7_API_KEY` |

Verification steps:

1. Confirm the plugin is enabled.
2. Open the MCP or tools section exposed by your VS Code build.
3. Confirm `io.github.upstash/context7` appears as an available server.
4. Start a chat that needs library documentation and allow the Context7 MCP server when prompted.
5. Provide `CONTEXT7_API_KEY` only through the secure VS Code input prompt, not in chat text.

Expected behavior:

- VS Code starts the configured MCP command when the server is used.
- The server requests `CONTEXT7_API_KEY` if it is not already available.
- The chat can retrieve library documentation through Context7.

If the server is unavailable, check that Node.js and `npx` are available on `PATH`, that your organization allows MCP usage, and that the API key prompt was completed.

## How to disable hooks if needed

Hooks can run local commands. Disable them if you need to inspect the plugin in a no-execution mode, debug hook failures, or comply with a restrictive environment.

Preferred safe options:

| Option | When to use it | Result |
| --- | --- | --- |
| Disable the plugin | You want to stop all plugin behavior. | Agents, commands, skills, MCP, and hooks stop loading from this plugin. |
| Use a local safety copy without hooks | You want agents and skills but no hook execution while testing. | The local copy can omit the manifest `hooks` entry before you register it with `chat.pluginLocations`. |
| Deny trust or policy prompts | You are not ready to allow local code execution. | VS Code should not run the plugin-provided local hook commands. |

Do not delete hook scripts just to disable execution. Prefer disabling the plugin or using a separate local copy so updates from the source repository remain reviewable.

## Troubleshooting

| Symptom | Likely cause | What to check |
| --- | --- | --- |
| Agent Plugins UI is missing | Agent Plugins preview is unavailable or disabled by policy. | Confirm your VS Code build supports Agent Plugins and check organization policy. |
| Plugin does not appear from `chat.pluginLocations` | Path points to the wrong folder or VS Code has not reloaded. | Point to the repository root that contains `.plugin/plugin.json`, then reload VS Code. |
| Commands are missing | Plugin is disabled or command assets did not load. | Confirm `.plugin/plugin.json` references `commands/` and the plugin is enabled. |
| `sdd-orchestrator` is missing | Agent assets did not load. | Confirm `.plugin/plugin.json` references `agents/` and the Agent Plugins view shows no errors. |
| Skills seem unavailable | Skill assets did not load or the request did not trigger a skill. | Confirm `.plugin/plugin.json` references `skills/` and retry with an SDD workflow request. |
| MCP server is missing | MCP is disabled, blocked by policy, or unavailable in the current build. | Check MCP/tool settings, organization policy, Node.js, and `npx`. |
| Context7 prompts for a key | `CONTEXT7_API_KEY` is required. | Provide it through the VS Code prompt when you trust the server invocation. |
| Hook execution fails | PowerShell execution, path resolution, or script policy issue. | Review `hooks.json`, `${PLUGIN_ROOT}` resolution, and the scripts under `scripts/hooks/`. |
| Hooks should not run | Local code execution is not allowed in the current environment. | Disable the plugin or use a local safety copy without hooks. |

## Update/version bump policy

Treat the plugin manifest version as the installed bundle version.

| Change type | Version guidance | Examples |
| --- | --- | --- |
| Patch | Bug fixes or docs that do not change plugin behavior. | Clarify install docs, fix typos, improve troubleshooting. |
| Minor | Backward-compatible plugin capability changes. | Add a command, add a skill, add a new hook with safe defaults, update MCP configuration without breaking existing use. |
| Major | Breaking plugin behavior or trust-surface changes. | Rename commands, remove agents, change required MCP inputs, replace hook behavior, or change SDD phase contracts incompatibly. |

Update flow for installed users:

1. Read the changelog or diff before updating, especially changes to `.plugin/plugin.json`, `.mcp.json`, `hooks.json`, and `scripts/hooks/`.
2. Pull or reinstall from the Git repository URL.
3. Reload VS Code.
4. Re-run the agent, command, MCP, and hook checks from this document.

When a release changes hooks or MCP servers, call that out explicitly because it changes the local execution and trust surface.
